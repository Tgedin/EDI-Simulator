# Documentation Principles - Quick Reference

## When to Document - 4 Triggers

1. **Made a choice between options?** → ADR (5 min, right now)
2. **Finished coding session?** → Journal entry (2 min, what + next)
3. **Completed a phase?** → Update README status
4. **"Aha!" moment or lesson learned?** → Journal "Learned" section ⭐

---

## What to Document - 4 Focus Areas

1. **Decisions:** WHY you chose X over Y (future you will forget)
2. **Progress:** WHAT works now, what's next (momentum tracker)
3. **Architecture:** HOW pieces connect (one diagram + brief text)
4. **Insights:** Unexpected lessons, gotchas, patterns discovered

---

## Journal Template

```markdown
## 2026-02-09

**Done:** Basic API working

**Learned:** 
- Docker volumes need explicit mapping for persistence
- PostgreSQL connection pooling matters even at small scale

**Wisdom:** "Start with sync endpoints, add async only when you feel the pain"

**Next:** Message generation
```

---

## Golden Rule

> "If you spent 10+ minutes choosing between approaches,  
> spend 5 minutes documenting why."

---

## Key Insight

**The best documentation isn't just WHAT you built, but WHAT YOU DISCOVERED while building it.**

Capture those insights - they're often the most valuable part for your growth as an architect.
